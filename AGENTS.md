# Language Library 项目工作规则

## 沟通与汇报

- 所有计划、解释、提问、进度汇报和最终报告使用中文。
- 命令、代码、路径、文件名和必要技术术语保留原文。
- 汇报必须基于实际检查结果，不得虚构测试、构建、部署或线上验收结果。

## 分支与 Git 规则

- `main` 是正式稳定分支，只保留已经验收的项目源码和正式记录。
- 所有修改必须从最新 `main` 创建独立功能分支或文档分支。
- 禁止直接在 `main` 上开发或提交修改。
- 禁止 `force push`、强制推送、改写 Git 历史、删除远程历史或使用破坏性回退操作。
- 合并回 `main` 必须使用安全的 fast-forward 流程；无法安全 fast-forward 时停止并报告。
- GitHub Pages 发布源为 `gh-pages` 分支根目录 `/`，不得擅自更换发布源。

## 原始资源保护

- `assets/` 中的原始音频和原始图片是源码母版，不得修改、删除、覆盖、重命名或重新编码。
- `data/` 中与原始资源相关的追溯字段不得因发布压缩而破坏。
- 压缩音频只允许作为 `dist/` 和 `gh-pages` 的发布资源使用，不得写回或替换 `assets/` 中的原始音频。
- 正式发布产物不得发布原书图片；原始图片继续保留在 `main` 源码中。

## 标准发布流程

任何会影响正式站的修改必须按以下顺序执行：

1. 从最新 `main` 创建功能分支并完成开发。
2. 在本地执行与任务匹配的检查和测试。
3. 使用 `tools/build_dist.py` 构建并验证 `dist/`。
4. 使用经过验证的 `dist/` 内容更新 `gh-pages`。
5. 等待 GitHub Pages 部署并完成真实线上验收。
6. 只有线上验收通过后，才允许使用 fast-forward 合并 `main`。

未通过线上验收前，不得合并 `main`，也不得把未验收版本作为正式版本发布。

## 开始任务前

- 必须检查当前分支、工作区状态和远程同步状态。
- 必须阅读项目根目录的 `AGENTS.md`、`README.md`、`TASK_STATUS.md` 以及当前任务相关报告。
- 必须确认任务范围，避免修改功能代码、`dist/` 或 `gh-pages` 中不属于当前任务的内容。
- 涉及正式发布时，必须先确认当前线上版本和发布源正常。

## 完成任务后

- 必须执行与修改范围匹配的测试、语法检查、引用检查或文档检查。
- 必须检查 `git diff --check` 和 `git status`。
- 最终汇报必须包含提交号、文件变化、实际测试结果、线上状态（如适用）和工作区状态。
- 如果存在未完成事项、无法验证的内容或风险，必须明确说明，不得将其描述为已完成。
